## ARQUITECTURAS DE SISTEMAS Y SERVICIOS WEB

## 1. ARQUITECTURA CLIENTE-SERVIDOR
Es un modelo de diseño de software en el que las tareas se reparten entre los proveedores de recursos o servicios (servidores) y los demandantes (clientes).
## Tipología según la carga (Distribución de cómputo):

* Cliente Ligero (Thin Client): El cliente solo gestiona la interfaz de usuario. Casi todo el procesamiento y el almacenamiento ocurren en el servidor (ej. un navegador web).
* Cliente Pesado (Fat/Rich Client): El cliente realiza gran parte del procesamiento y validaciones locales. El servidor se limita a la gestión de datos.
* Arquitectura de 2 capas: La relación es directa entre el cliente y el servidor (típico en aplicaciones de escritorio antiguas conectadas a una base de datos).

## Componentes:

   1. Cliente: Aplicación que inicia peticiones.
   2. Servidor: Proceso que escucha, procesa y responde.
   3. Middleware: Capa de software que facilita la comunicación entre ambos.
   4. Protocolo: Reglas que rigen el intercambio (ej. HTTP, TCP/IP).

------------------------------
## 2. ARQUITECTURA MULTICAPAS (N-TIER)
Evolución de la anterior que busca la separación de responsabilidades, dividiendo el sistema en niveles lógicos independientes:

   1. Capa de Presentación: Interfaz de usuario (HTML, App móvil). Se comunica solo con la capa de negocio.
   2. Capa de Negocio (Lógica): Donde se procesan las reglas de la aplicación y los cálculos. Es el "cerebro".
   3. Capa de Datos: Gestiona el almacenamiento (SGBD). Solo responde a peticiones de la capa de negocio.

------------------------------
## 3. VENTAJAS E INCONVENIENTES

| Ventajas | Inconvenientes |
|---|---|
| Centralización: Fácil mantenimiento y seguridad de los datos en el servidor. | Punto único de fallo: Si el servidor cae, ningún cliente funciona. |
| Escalabilidad: En multicapas, puedes mejorar solo el servidor de datos sin tocar la interfaz. | Congestión: Un exceso de peticiones puede saturar el servidor (cuello de botella). |
| Integridad: Las reglas de negocio se aplican en un solo lugar. | Complejidad: Más difícil de programar y depurar que un sistema monolítico. |

------------------------------
## 4. ARQUITECTURA DE SERVICIOS WEB
Permite la interacción entre máquinas a través de una red mediante estándares que garantizan la interoperabilidad (independientemente del lenguaje).
## Tipos principales:

* SOAP (Simple Object Access Protocol):
* Basado en XML.
   * Protocolo estricto, con altos estándares de seguridad y transaccionalidad.
   * Usa el estándar WSDL para describir sus servicios.
* REST (Representational State Transfer):
* No es un protocolo, sino un estilo arquitectónico.
   * Usa los métodos estándar de HTTP (GET, POST, PUT, DELETE).
   * Mucho más ligero, utiliza preferiblemente JSON.
   * Es la base de la web moderna y las APIs móviles.

## Microservicios:
Es la tendencia actual donde una aplicación se divide en pequeños servicios web independientes que se comunican entre sí. Si un microservicio falla, el resto de la aplicación puede seguir funcionando.

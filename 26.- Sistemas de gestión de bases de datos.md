# SISTEMAS DE GESTIÓN DE BASES DE DATOS (SGBD / DBMS)

## 1. DEFINICIÓN Y CONCEPTOS

Un SGBD es un software que actúa como interfaz entre la base de datos, el usuario y las aplicaciones. Su objetivo es permitir la creación, recuperación, actualización y gestión de los datos de forma segura, íntegra y eficiente.

* Independencia de datos: Los datos son independientes de las aplicaciones que los usan.
* Integridad: Reglas que aseguran que los datos sean correctos (ej. no permitir una edad negativa).
* Seguridad: Control de quién puede ver o modificar cada dato.

## 2. ARQUITECTURA (MODELO ANSI/SPARC)

Se divide en tres niveles para separar la visión del usuario de la realidad física:

   1. Nivel Externo (Vistas): Lo que ve cada usuario según sus permisos.
   2. Nivel Conceptual (Esquema): La estructura lógica global (qué tablas y relaciones existen).
   3. Nivel Interno (Físico): Cómo se almacenan realmente los bits en el disco.

## 3. BASES DE DATOS RELACIONALES (SQL)

Es el modelo tradicional basado en tablas relacionadas mediante claves.

* Características:
  * Esquema rígido: La estructura debe definirse antes de insertar datos.
  * Propiedades ACID: Garantizan que las transacciones sean seguras (Atomicidad, Consistencia, Aislamiento y Durabilidad).
  * Lenguaje: Utilizan SQL (Structured Query Language).
* Componentes: Tablas (entidades), Filas (registros/tuplas), Columnas (campos/atributos) y Claves (Primarias y Foráneas).
* Ejemplos: Oracle, MySQL, PostgreSQL, SQL Server.

## 4. BASES DE DATOS NO RELACIONALES (NoSQL)

Diseñadas para grandes volúmenes de datos, alta velocidad y estructuras flexibles.

* Características:
  * Esquema flexible (Schema-less): Puedes guardar datos sin una estructura fija previa.
  * Escalabilidad horizontal: Muy fáciles de ampliar añadiendo más servidores.
  * Teorema CAP: Priorizan la Disponibilidad y la Tolerancia a particiones frente a la Consistencia inmediata (Consistencia eventual).
* Tipología:
  * Documentales: Guardan datos en formato JSON/BSON (ej. MongoDB).
  * Clave-Valor: Diccionarios ultra rápidos (ej. Redis).Este bloque aborda el corazón del almacenamiento estructurado y moderno de las organizaciones: los SGBD (Sistemas de Gestión de Bases de Datos).

## 5. ESTÁNDARES DE CONECTIVIDAD

Son las "clavijas" universales que permiten a una aplicación conectar con cualquier base de datos:

* ODBC (Open Database Connectivity): Estándar abierto creado por Microsoft que permite que aplicaciones C/C++ conecten con cualquier SGBD usando un driver específico.
* JDBC (Java Database Connectivity): El estándar equivalente para aplicaciones escritas en Java.
* ADO.NET: La evolución de Microsoft para el entorno .NET.
* APIs REST: Común en bases de datos NoSQL para interactuar mediante HTTP/JSON.

Dato clave: En entornos modernos se utiliza la Persistencia Políglota, que consiste en usar una base de datos SQL para transacciones críticas (pagos) y una NoSQL para datos masivos o búsquedas rápidas (catálogos, sesiones).

Esta comparativa es fundamental para entender por qué elegimos un sistema u otro dependiendo de si priorizamos la seguridad total de la transacción o la disponibilidad masiva del servicio.

## Comparativa: Propiedades ACID vs. BASE

| Característica | ACID (Tradicional / SQL) | BASE (Moderno / NoSQL) |
| --- | --- | --- |
| Enfoque | Prioriza la Consistencia. | Prioriza la Disponibilidad. |
| Siglas | Atomicity, Consistency, Isolation, Durability. | Basic Availability, Soft-state, Eventual consistency. |
| Significado | O se realiza la transacción completa o no se hace nada. El dato es real al instante. | El sistema funciona aunque haya fallos. El dato puede tardar unos segundos en actualizarse en todos los nodos. |
| Consistencia | Inmediata: Todos los usuarios ven lo mismo al mismo tiempo. | Eventual: Los datos se sincronizarán eventualmente (en milisegundos o segundos). |
| Escalabilidad | Principalmente Vertical (necesitas un servidor más potente). | Horizontal (añades más servidores baratos en clúster). |
| Uso ideal | Sistemas financieros, inventarios, nóminas, registros legales. | Redes sociales, carritos de compra, Big Data, análisis en tiempo real. |

## Resumen para tu documento

   1. Modelo ACID: Es como una transferencia bancaria. Si el dinero sale de tu cuenta pero no llega a la otra por un fallo, el sistema deshace todo (Rollback) para que no se pierda nada. Es rígido y seguro.
   2. Modelo BASE: Es como un post en una red social. Si publicas una foto, puede que un amigo en Japón tarde 2 segundos más en verla que un amigo en tu misma ciudad, pero el sistema nunca deja de funcionar. Es flexible y veloz.

Dato extra: El Teorema CAP afirma que en un sistema distribuido solo puedes garantizar dos de estas tres propiedades: Consistencia, Ata disponibilidad o tolerancia a Particiones.
